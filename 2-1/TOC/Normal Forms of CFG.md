# Normal Forms for Context-Free Grammars

## Introduction to Normal Forms


![[Chapter-7-1.webp]]

### What are Normal Forms?

A **normal form** is a standardized way of writing grammar productions that follows specific restrictions while preserving the language generated.

**Key Concept:** Every context-free grammar (that doesn't generate ε) can be transformed into an equivalent grammar in **Chomsky Normal Form** or **Greibach Normal Form**.

**"Equivalent" means:**

- Both grammars generate exactly the same language
- L(G₁) = L(G₂)
- Different structure, same strings

**Intuitive Analogy:** Think of normal forms like standardized formats for mathematical expressions. Just as we can write fractions in "lowest terms" or polynomials in "standard form," we can write grammars in standard forms. The mathematical meaning stays the same, but the representation becomes simpler and more predictable.

### Why Normal Forms Matter

**Theoretical Importance:**

- Simplifies proofs about CFGs
- Makes analysis easier
- Reveals structural properties of languages
- Enables algorithmic approaches

**Practical Importance:**

- **Parsing algorithms** - especially CYK (Cocke-Younger-Kasami) algorithm
- **Complexity analysis** - polynomial-time string membership testing
- **Compiler design** - predictable grammar structure
- **Language recognition** - systematic approach

**Example Use Case - CYK Algorithm:**

- Given: A grammar in Chomsky Normal Form
- Given: A string w
- Question: Is w ∈ L(G)?
- CYK answers in **O(n³)** time (polynomial!)
- Without CNF, much harder!

### The Two Main Normal Forms

**1. Chomsky Normal Form (CNF):**

- All productions are either:
    - A → BC (two variables)
    - A → a (single terminal)
- Very restricted, very useful
- Named after linguist Noam Chomsky

**2. Greibach Normal Form (GNF):**

- All productions are:
    - A → aα (terminal followed by variables)
- Useful for PDA construction
- Named after Sheila Greibach

**This chapter focuses primarily on Chomsky Normal Form.**

## Properties of Context-Free Languages

![[Chapter-7-2.webp]]
![[Chapter-7-3.webp]]

### Equivalent Definitions of CFLs

**Three equivalent characterizations:**

1. **Grammar-based:** Languages generated by CFGs
2. **Automata-based:** Languages accepted by PDAs (non-deterministic)
3. **Set-based:** Sets of terminal sequences accepted by the grammar

**Fundamental Theorem:** L is context-free ⟺ L = L(G) for some CFG ⟺ L = N(P) for some PDA

### Closure Properties

**Closed under (always produces CFL):**

✓ **Union:** If L₁ and L₂ are CFLs, then L₁ ∪ L₂ is a CFL

- Proof: Create new start symbol S → S₁ | S₂

✓ **Concatenation:** If L₁ and L₂ are CFLs, then L₁L₂ is a CFL

- Proof: Create new start symbol S → S₁S₂

✓ **Kleene Star:** If L is a CFL, then L* is a CFL

- Proof: S → SS | S₁ | ε

✓ **Reversal:** If L is a CFL, then Lᴿ is a CFL

- Proof: Reverse all production bodies

**NOT closed under (may not be CFL):**

✗ **Intersection:** L₁ ∩ L₂ may not be CFL even if both are CFLs

- Example: {aⁿbⁿcⁿ} can be expressed as intersection of two CFLs
- But {aⁿbⁿcⁿ} itself is not context-free!

✗ **Complement:** L̄ may not be CFL even if L is CFL

- Related to intersection via De Morgan's laws
- Some complements are CFL, some aren't

### Special Properties

**Regular languages ⊂ CFLs:**

- Every regular language is context-free
- Can be described by a regular grammar (which is also a CFG)
- Proof: Any DFA can be converted to a CFG

**Intersection with Regular:**

- If L₁ is CFL and L₂ is regular, then **L₁ ∩ L₂ is always CFL**
- Very useful property!
- Can be proven using PDA + DFA construction

**Context-Sensitive Languages:**

- There exist context-sensitive languages that are NOT context-free
- Language hierarchy: Regular ⊂ CFL ⊂ Context-Sensitive ⊂ Recursively Enumerable

### Testing and Decidability

**Pumping Lemma for CFLs:**

- Used to prove a language is NOT context-free
- Similar to pumping lemma for regular languages, but more complex
- Essential tool for showing CFL limitations

**Undecidable Problem:**

- Given a context-sensitive grammar, is it actually context-free?
- **This problem is undecidable!** (cannot be solved algorithmically)
- Shows fundamental limits of what we can compute about grammars

## Goal of Normal Forms

![[Chapter-7-4.webp]]

### The Target: Chomsky Normal Form

**Objective:** Transform any CFL (without ε) into a grammar where all productions are either:

**Form 1:** A → BC

- A, B, C are all variables
- Body has exactly two variables

**Form 2:** A → a

- A is a variable
- a is a terminal
- Body has exactly one terminal

**Why these forms?**

- Extremely simple and predictable
- Easy to analyze and parse
- Every derivation tree has a specific structure
- Enables efficient algorithms

**What we gain:**

- Predictable parse tree shape (binary tree)
- Bounded derivation step size
- Systematic parsing approaches
- Polynomial-time algorithms

**What we lose:**

- Nothing! The language stays the same
- Only the grammar structure changes

---

## Simplification Steps Overview

![[Chapter-7-5.webp]]

### Three Required Simplifications

To achieve Chomsky Normal Form, we need three elimination steps:

**1. Eliminate Useless Symbols**

- Remove variables/terminals that don't appear in any derivation
- Two aspects: generating and reachable
- Cleans up the grammar

**2. Eliminate ε-Productions**

- Remove productions of the form A → ε
- Except we may lose ε from the language
- Result: L(G₁) = L(G) - {ε}

**3. Eliminate Unit Productions**

- Remove productions of the form A → B (variable to variable)
- These just add unnecessary derivation steps
- Can be eliminated without changing the language

**Order matters!** Must be done in specific sequence to avoid reintroducing eliminated elements.

**Intuitive Understanding:** Think of this like cleaning and organizing a messy room:

1. First, throw out trash (useless symbols)
2. Then, remove empty boxes (ε-productions)
3. Finally, eliminate redundant containers (unit productions) Only then can we reorganize into the neat structure (CNF)!

## Eliminating Useless Symbols

![[Chapter-7-6.webp]]
![[Chapter-7-7.webp]]

### Definition of Useful Symbols

**Symbol X is useful for G = (V, T, P, S) if:**

There exists a derivation: __S ⇒_ αXβ ⇒_ w** where w ∈ T*

**Breaking it down:**

- Start symbol S derives a sentential form containing X
- That sentential form (containing X) eventually derives a terminal string
- X contributes to generating at least one string in the language

**Symbol X can be:**

- A variable (X ∈ V), or
- A terminal (X ∈ T)

**Two-part requirement:**

1. __S ⇒_ αXβ_* - X must be reachable from the start symbol
2. __αXβ ⇒_ w_* - X must help generate a terminal string

**If X is not useful, then X is useless!**

### Two Properties of Useful Symbols

**INSERT PAGE 8 OF PDF HERE**

**Property 1 - Generating:**

- X is **generating** if X ⇒* w for some terminal string w
- Can eventually produce a string of terminals
- Every terminal is automatically generating (derives itself in 0 steps)

**Property 2 - Reachable:**

- X is **reachable** if S ⇒* αXβ for some α and β
- Can be reached from the start symbol
- S itself is automatically reachable

**Key Theorem:** **X is useful ⟺ X is both generating AND reachable**

**Why both are necessary:**

- **Generating but not reachable:** X can produce strings, but we can never get to X from S → useless!
- **Reachable but not generating:** We can get to X, but X can't produce terminals → useless!

**Example:**

```
S → AB | a
A → BC
B → b
C → c
D → d
```

- S, A, B, C are reachable (can derive from S)
- S, B, C, D are generating (can derive terminals)
- D is generating but not reachable → useless!

## Elimination Algorithm

![[Chapter-7-9.webp]]

### Strategy

**Critical order of elimination:**

1. **First:** Eliminate non-generating symbols
2. **Second:** Eliminate non-reachable symbols (from remaining grammar)

**Why this order?**

- If we eliminate non-reachable first, we might leave generating symbols that later become non-generating
- Doing generating first, then reachable, guarantees only useful symbols remain

**Result:** A grammar with only useful symbols!

## Computing Generating Symbols

![[Chapter-7-13.webp]]

### Algorithm

**Basis:**

- Every symbol in T is generating
- Terminals generate themselves (0 derivation steps)

**Induction:**

- If production A → α exists
- And every symbol in α is already known to be generating
- Then A is generating

**Special case:** If A → ε, then A is generating

- ε is the "empty terminal string"
- A derives ε in one step

### Theorem

The algorithm finds **all and only** the generating symbols of G.

**Proof idea:**

- **(Only)** If algorithm marks A as generating, then A ⇒* w for some w ∈ T*
    - By induction on when A was marked
- **(All)** If A ⇒* w for some w ∈ T*, then algorithm marks A
    - By induction on length of derivation

**Example trace:**

```
Grammar:
S → AB | a
A → BC | b
B → b
C → c
D → d | DE
E → e

Round 1: Mark terminals: a, b, c, d, e ✓
Round 2: Mark A (A → b, b is generating)
         Mark B (B → b, b is generating)
         Mark C (C → c, c is generating)
         Mark D (D → d, d is generating)
         Mark E (E → e, e is generating)
Round 3: Mark S (S → a, a is generating; S → AB, both generating)

All symbols are generating!
```

## Computing Reachable Symbols

![[Chapter-7-14.webp]]

### Algorithm

**For grammar G = (V, T, P, S):**

**Basis:**

- S is reachable (start symbol)

**Induction:**

- If variable A is discovered to be reachable
- Then for all productions A → α
- All symbols in α are also reachable

**Process:**

1. Mark S as reachable
2. For each marked variable, look at all its production bodies
3. Mark all symbols in those bodies as reachable
4. Repeat until no new symbols are marked

### Theorem

The algorithm finds **all and only** the reachable symbols of G.

**Example trace:**

```
Grammar:
S → AB | a
A → b
B → c
D → d

Round 1: S is reachable (basis)
Round 2: From S → AB | a:
         Mark A, B, a as reachable
Round 3: From A → b:
         Mark b as reachable
         From B → c:
         Mark c as reachable
Round 4: No new symbols (D never reached)

Reachable: S, A, B, a, b, c
Not reachable: D
```

## Summary: Useless Symbol Elimination

![[Chapter-7-15.webp]]

### Complete Process

**Step 1:** Eliminate non-generating symbols

- Run generating symbol algorithm
- Remove all productions containing non-generating symbols
- Remove all non-generating variables from V

**Step 2:** Eliminate non-reachable symbols (from resulting grammar)

- Run reachable symbol algorithm
- Remove all non-reachable symbols and their productions

**Result:** Grammar with only useful symbols!

**Important:** Order matters!

- Generating first, then reachable ✓
- Reachable first, then generating ✗ (doesn't work)

## Eliminating ε-Productions

![[Chapter-7-16.webp]]![[Chapter-7-17.webp]]

### Strategy and Goal

**Goal:** If L is a CFL, then **L - {ε}** is also a CFL

**Strategy:**

1. Find all "nullable" variables (those that can derive ε)
2. For each production, create versions with/without nullable variables
3. Remove all ε-productions

**Key Concept - Nullable Variable:**

- Variable A is **nullable** if A ⇒* ε
- Can derive the empty string in zero or more steps

**Why this matters:**

- If A is nullable and appears in a production body B → αAβ
- Then we need both B → αAβ (A doesn't vanish) and B → αβ (A does vanish)
- This handles all cases where A might or might not derive ε

### Finding Nullable Variables

![[Chapter-7-18.webp]]

**Basis:**

- If A → ε is a production, then A is nullable
- Direct one-step derivation to ε

**Induction:**

- If production B → C₁C₂...Cₖ exists
- Where each Cᵢ is a variable
- And each Cᵢ is nullable
- Then B is nullable

**Why induction works:**

- If all symbols in the body can derive ε
- Then the whole body can derive ε (C₁ ⇒* ε, C₂ ⇒* ε, etc.)
- So B ⇒ C₁C₂...Cₖ ⇒* ε

**Example:**

```
Grammar:
S → AB
A → aAb | ε
B → bB | ε

Round 1: A is nullable (A → ε)
         B is nullable (B → ε)
Round 2: S is nullable (S → AB, both A and B are nullable)

All variables are nullable!
```

---

## Theorem: Correctness of Nullable Algorithm

![[Chapter-7-19.webp]]

### Statement

For any grammar G, the only nullable symbols are the variables found by the nullable algorithm.

### Proof

**By induction on the number of steps** in the derivation A ⇒* ε:

**Basis (1 step):**

- If A ⇒ ε in one step
- Then A → ε must be a production
- Algorithm discovers A as nullable (basis case) ✓

**Induction (N > 1 steps):**

- If A ⇒* ε in N steps
- First step must be A ⇒ C₁C₂...Cₖ
- Then C₁C₂...Cₖ ⇒* ε in < N steps
- Each Cᵢ ⇒* ε individually in < N steps
- By inductive hypothesis, each Cᵢ is discovered as nullable
- Then algorithm discovers A as nullable (inductive case) ✓

**Conclusion:** Algorithm finds exactly the nullable variables.

## Constructing Grammar Without ε-Productions

![[Chapter-7-20.webp]]

### Construction Algorithm

**For each production A → X₁X₂...Xₖ in G:**

1. Identify which Xᵢ are nullable variables
2. Create all versions with different subsets of nullable variables present/absent
3. Add all these versions to new grammar (except A → ε)

**Example:**

```
Production: A → BCD
If B and D are nullable, but C is not:

Generate:
A → BCD  (all present)
A → BC   (D absent)
A → CD   (B absent)  
A → C    (B and D absent)
```

**Do NOT add:** A → ε even if original had it (unless A is start symbol and we want ε in language)

### Theorem

If G₁ is constructed by eliminating ε-productions from G, then: **L(G₁) = L(G) - {ε}**

**Key points:**

- All non-ε strings derivable in G are still derivable in G₁
- ε is removed (unless special handling for start symbol)
- No other strings are added or removed

## Eliminating Unit Productions

![[Chapter-7-21.webp]]
![[Chapter-7-22.webp]]
### What are Unit Productions?

**Definition:** A production of the form **A → B** where both A and B are variables is called a **unit production**.

**Why eliminate them?**

- They introduce extra derivation steps
- Don't generate any terminals
- Not needed for the language
- Make grammar longer than necessary

**Example of unnecessary steps:**

```cpp
S → A    (unit)
A → B    (unit)
B → a    (non-unit)

Derivation: S ⇒ A ⇒ B ⇒ a
Could be shortened to: S ⇒ a
```

### Unit Pairs

![[Chapter-7-23.webp]]

**Definition:** (A, B) is a **unit pair** if A ⇒* B using only unit productions.

**Computing unit pairs:**

**Basis:**

- (A, A) is a unit pair for every variable A
- Reflexive: every variable reaches itself in 0 steps

**Induction:**

- If (A, B) is a unit pair
- And B → C is a production (where C is a variable)
- Then (A, C) is a unit pair

**Theorem:** This algorithm finds exactly all unit pairs for grammar G.

**Example:**

```
Grammar:
S → A
A → B | a
B → C | b
C → c

Unit pairs:
(S, S) - basis
(A, A) - basis
(B, B) - basis
(C, C) - basis
(S, A) - from S → A
(A, B) - from A → B
(B, C) - from B → C
(S, B) - from (S, A) and A → B
(A, C) - from (A, B) and B → C
(S, C) - from (S, B) and B → C
```

## Summary of Three Eliminations

![[Chapter-7-27.webp]]

### Theorem

If G is a CFG generating a language with at least one string (possibly including ε), then there exists CFG G₁ such that:

**Properties of G₁:**

1. **L(G₁) = L(G) - {ε}** - same language except possibly ε
2. **No ε-productions** - (except possibly S → ε if S doesn't appear in any body)
3. **No unit productions** - no A → B
4. **No useless symbols** - all symbols are both generating and reachable

**This is the prepared grammar ready for conversion to Chomsky Normal Form!**

## Proof Strategy

![[Chapter-7-28.webp]]

### Order of Operations

**Step 1:** Eliminate ε-productions

- Results in grammar with no ε-productions (except possibly S → ε)

**Step 2:** Eliminate unit productions

- Won't reintroduce ε-productions
- Why? New production bodies come from old grammar, which had no ε-productions

**Step 3:** Eliminate useless symbols

- Only deletes productions and symbols
- Never adds anything back
- Can't reintroduce ε-productions or unit productions

**Critical insight:** Each step preserves the work of previous steps!

## Chomsky Normal Form Definition

![[Chapter-7-29.webp]]

### Formal Definition

A grammar G is in **Chomsky Normal Form (CNF)** if every production is in one of these forms:

**Form 1:** A → BC

- A, B, C are all variables
- Body has exactly two variables

**Form 2:** A → a

- A is a variable
- a is a terminal
- Body is a single terminal

**Additionally:**

- G has no useless symbols
- (Optionally: S → ε if ε ∈ L(G), and S doesn't appear in any production body)

**What's NOT allowed in CNF:**

- ✗ Productions like A → BCD (three or more variables)
- ✗ Productions like A → aB (mixing terminals and variables)
- ✗ Productions like A → ab (multiple terminals)
- ✗ Productions like A → B (unit productions)
- ✗ Productions like A → ε (except possibly for start symbol)

## Converting to Chomsky Normal Form

![[Chapter-7-30.webp]]

### Two-Phase Construction

**Prerequisite:** Grammar with no ε-productions, unit productions, or useless symbols

**Phase 1: Arrange Long Bodies**

- Make all bodies of length ≥ 2 contain only variables
- Replace terminals in mixed bodies with new variables

**Phase 2: Break Long Bodies**

- Break bodies of length ≥ 3 into cascade of binary productions
- Each production has exactly 2 variables in body

### Phase 1 - Replace Terminals in Long Bodies

**For each terminal a appearing in a body of length ≥ 2:**

1. Create new variable Uₐ
2. Add production Uₐ → a
3. Replace all occurrences of a in bodies of length ≥ 2 with Uₐ

**Example:**

```
Before:
A → aBC
B → bC
C → c

After Phase 1:
A → UₐBC    (replaced a)
B → UᵦC     (replaced b)
C → c       (length 1, no change)
Uₐ → a      (new)
Uᵦ → b      (new)
```

**Result:** All bodies of length ≥ 2 contain only variables!

### Phase 2 - Break Long Bodies into Binary

**For each production A → B₁B₂B₃...Bₖ where k ≥ 3:**

1. Create new variables D₁, D₂, ..., Dₖ₋₂
2. Replace with cascade:
    
    ```
    A → B₁D₁D₁ → B₂D₂D₂ → B₃D₃...Dₖ₋₂ → Bₖ₋₁Bₖ
    ```
    

**Example:**

```
Before:
A → BCDE

After Phase 2:
A → BD₁     (first two)
D₁ → CD₂    (next two)
D₂ → DE     (last two)
```

**Result:** All productions are either A → BC or A → a!

## Properties of CNF Grammars

### Parse Tree Structure

**Every CNF derivation tree is a binary tree!**

**Why?**

- Each variable has either:
    - 2 children (from A → BC production)
    - 1 child (from A → a production, and child is a leaf)

**Example derivation for "aabb":**

```
S → UₐB
  → Uₐ(UₐD₂)
  → Uₐ(Uₐ(BB))
  → Uₐ(Uₐ(UᵦB))
  → Uₐ(Uₐ(Uᵦ(Uᵦ)))
  → Uₐ(Uₐ(Uᵦ(b)))
  → Uₐ(Uₐ(b)(b))
  → Uₐ(a)(b)(b)
  → a(a)(b)(b)
  = aabb
```

**Tree structure:**

```
       S
      / \
     Uₐ  B
     |  / \
     a Uₐ D₂
       |  / \
       a  B  B
         / \ |
        Uᵦ  | Uᵦ
        |   | |
        b   Uᵦ b
            |
            b
(simplified)
```

### Derivation Length

**For a string of length n:**

- Exactly **2n - 1** derivation steps
- Each step either:
    - Replaces 1 variable with 2 variables (expanding)
    - Replaces 1 variable with 1 terminal (terminating)

**Why 2n - 1?**

- Start with 1 variable (S)
- Need to generate n terminals
- Each expanding step adds 1 variable (net +1 variable)
- Each terminating step removes 1 variable, adds 1 terminal (net +1 terminal)
- Need (n-1) expanding steps and n terminating steps
- Total: (n-1) + n = 2n - 1

## Applications of CNF

### CYK Algorithm (Cocke-Younger-Kasami)

**Problem:** Given string w and CFG G in CNF, is w ∈ L(G)?

**CYK Algorithm:**

- Uses dynamic programming
- Builds table of "which variables can generate which substrings"
- **Time complexity: O(n³·|G|)** where n = |w|
- Polynomial time! Very efficient.

**How it works:**

1. Build table T where T[i,j] = set of variables that can derive substring w[i...j]
2. Base case: T[i,i] = {A | A → wᵢ is a production}
3. Inductive case: T[i,j] = {A | A → BC where B ∈ T[i,k] and C ∈ T[k+1,j] for some k}
4. Accept if S ∈ T[1,n]

**Example for w = "aabb":**

```
     a    a    b    b
  ┌────┌────┌────┌────┐
a │ Uₐ │    │    │    │
  ├────┼────┼────┼────┤
a │    │ Uₐ │    │    │
  ├────┼────┼────┼────┤
b │    │    │ Uᵦ │    │
  ├────┼────┼────┼────┤
b │    │    │    │ Uᵦ │
  └────┴────┴────┴────┘

(Fill in larger substrings using CNF productions)
```

**Why CNF is essential:**

- Productions have exactly 2 variables in body
- Makes dynamic programming straightforward
- Each cell depends on pairs of other cells
- Without CNF, algorithm much more complex!

## Summary and Key Takeaways

### The Complete Process

**Starting point:** Any CFG G

**Goal:** Equivalent grammar in Chomsky Normal Form

**Steps:**

1. ✓ Eliminate useless symbols (generating, then reachable)
2. ✓ Eliminate ε-productions (find nullable variables)
3. ✓ Eliminate unit productions (find unit pairs)
4. ✓ Arrange terminals (create Uₐ → a for terminals in long bodies)
5. ✓ Break long bodies (cascade into binary productions)

**Result:** Grammar in CNF with L(G_CNF) = L(G) - {ε}

### Chomsky Normal Form Properties

**Production forms:**

- A → BC (exactly two variables)
- A → a (exactly one terminal)

**Benefits:**

- ✓ Binary parse trees
- ✓ Predictable derivation length (2n-1 for string of length n)
- ✓ Enables CYK algorithm (O(n³) parsing)
- ✓ Simplifies proofs and analysis
- ✓ No ambiguity in production form

**Restrictions:**

- ✗ Cannot include ε (except special case for S)
- ✗ All productions highly structured
- ✗ May have many more productions than original
- ✗ May have many new variables (Uₐ, D₁, D₂, etc.)

### Important Theorems

**Theorem 1:** Every CFL without ε has an equivalent CNF grammar

**Theorem 2:** Useless symbol elimination preserves language

**Theorem 3:** ε-production elimination yields L(G₁) = L(G) - {ε}

**Theorem 4:** Unit production elimination preserves language

**Theorem 5:** CNF enables polynomial-time parsing (CYK algorithm)

## Detailed Example Walkthrough

### Complete Transformation Example

**Starting grammar:**

```
S → ASB | ε | b
A → aA | a | ε
B → bB | A
```

### Step 1: Eliminate Useless Symbols

**Find generating:**

- Terminals: a, b (generating)
- A generates a (A → a) ✓
- B generates via B → A, and A generates ✓
- S generates b (S → b) ✓
- All symbols are generating!

**Find reachable:**

- S is reachable (start)
- A, B, a, b are all reachable from S
- All symbols are reachable!

**Result:** No useless symbols, grammar unchanged.

### Step 2: Eliminate ε-Productions

**Find nullable:**

- Round 1: S → ε (S is nullable), A → ε (A is nullable)
- Round 2: B → A and A is nullable (B is nullable)
- Nullable variables: {S, A, B}

**Create new productions:**

From S → ASB:

- S → ASB (all present)
- S → SB (A absent)
- S → AS (B absent)
- S → S (both absent) - this is a unit production

From S → ε:

- Don't add (removing ε-productions)

From S → b:

- S → b (no nullable variables)

From A → aA:

- A → aA (A present)
- A → a (A absent)

From A → a:

- A → a (no nullable variables)

From A → ε:

- Don't add

From B → bB:

- B → bB (B present)
- B → b (B absent)

From B → A:

- Don't add (unit production, A is nullable)

**Grammar after ε-elimination:**

```
S → ASB | SB | AS | S | b
A → aA | a
B → bB | b
```

### Step 3: Eliminate Unit Productions

**Find unit pairs:**

- (S, S), (A, A), (B, B) - reflexive
- (S, S) from S → S - already have

No other unit productions! (S → S will be removed)

**Result:**

```
S → ASB | SB | AS | b
A → aA | a
B → bB | b
```

### Step 4: Introduce Terminal Variables

**For bodies of length ≥ 2:**

From S → ASB:

- S → ASB (all variables, no change)

From S → SB:

- S → SB (all variables, no change)

From S → AS:

- S → AS (all variables, no change)

From A → aA:

- Need to replace 'a' with Uₐ
- A → UₐA

From B → bB:

- Need to replace 'b' with Uᵦ
- B → UᵦB

**Add new productions:**

```
Uₐ → a
Uᵦ → b
```

**Grammar after terminal replacement:**

```
S → ASB | SB | AS | b
A → UₐA | a
B → UᵦB | b
Uₐ → a
Uᵦ → b
```

### Step 5: Break Long Bodies

**From S → ASB (length 3):**

```
S → AD₁
D₁ → SB
```

**All other productions have length ≤ 2, no changes needed.**

### Final CNF Grammar

```
S → AD₁ | SB | AS | b
A → UₐA | a
B → UᵦB | b
D₁ → SB
Uₐ → a
Uᵦ → b
```

**Verification:**

- ✓ All productions are A → BC or A → a
- ✓ No ε-productions
- ✓ No unit productions
- ✓ No useless symbols
- ✓ Grammar is in Chomsky Normal Form!

## Study Tips and Practice

### To Master Normal Forms:

1. **Understand the order:**
    
    - Useless symbols (generating THEN reachable)
    - ε-productions
    - Unit productions
    - Terminal arrangement
    - Body breaking
    - **Order matters!**
2. **Practice each elimination:**
    
    - Work through examples step-by-step
    - Verify language is preserved
    - Check that eliminated elements don't return
3. **Recognize production forms:**
    
    - Identify which productions need conversion
    - Know which are already in CNF
    - Count variables in bodies carefully
4. **Trace derivations:**
    
    - Practice deriving strings in CNF grammar
    - Count derivation steps (should be 2n-1)
    - Draw binary parse trees
5. **Understand applications:**
    
    - Know why CNF is useful (CYK algorithm)
    - Understand time complexity benefits
    - Recognize when CNF is appropriate

### Common Mistakes to Avoid:

❌ **Wrong order of elimination** (reachable before generating) ❌ **Forgetting to remove all ε-productions** (except special S → ε) ❌ **Missing nullable variables** (not following inductive closure) ❌ **Missing unit pairs** (not finding transitive closure) ❌ **Creating unit productions** during ε-elimination ❌ **Replacing terminals in single-terminal bodies** (don't!) ❌ **Breaking bodies of length 2** (already correct!) ❌ **Not creating enough new variables** when breaking long bodies

### Key Questions to Ask:

✓ Is this symbol generating? ✓ Is this symbol reachable? ✓ Which variables are nullable? ✓ What are all the unit pairs? ✓ Does this body need terminal variables? ✓ Is this body too long? ✓ Is the final grammar in CNF? ✓ Did I preserve the language?

### Practice Problems:

**Level 1 - Basic:**

1. Identify useless symbols in given grammar
2. Find all nullable variables
3. Find all unit pairs
4. Determine if grammar is in CNF

**Level 2 - Intermediate:** 5. Eliminate ε-productions from given grammar 6. Eliminate unit productions from given grammar 7. Convert simple grammar to CNF

**Level 3 - Advanced:** 8. Complete CNF conversion from scratch 9. Prove language is preserved at each step 10. Use CYK algorithm to parse string with CNF grammar

### Verification Checklist:

After converting to CNF, check:

✓ **Every production is A → BC or A → a** ✓ **No production has A → BCD** (3+ variables) ✓ **No production has A → aB** (mixed) ✓ **No production has A → ab** (2+ terminals) ✓ **No production has A → B** (unit) ✓ **No production has A → ε** (unless special case) ✓ **All symbols are useful** (both generating and reachable) ✓ **Language is preserved** (can derive same strings)

## Complexity Analysis

### Size of CNF Grammar

**Original grammar:** n productions

**After conversion:**

- Worst case: **O(n²)** productions
- Why? Each nullable variable can double production count
- New variables: O(n) for terminals, O(n) for breaking bodies

**Example:**

```
Original: S → a₁a₂a₃...aₙ (one production)

CNF: 
S → Uₐ₁D₁
D₁ → Uₐ₂D₂
D₂ → Uₐ₃D₃
...
Dₙ₋₂ → Uₐₙ₋₁Uₐₙ
Uₐ₁ → a₁
Uₐ₂ → a₂
...
Uₐₙ → aₙ

Total: (n-1) + n = 2n-1 productions from one!
```

### Parsing Complexity

**Without CNF:**

- General CFG parsing: **O(n³)** to **exponential**
- Depends on grammar structure
- May require backtracking

**With CNF (CYK algorithm):**

- **Always O(n³·|G|)**
- |G| = size of grammar (constant for fixed G)
- Effectively **O(n³)** for parsing
- No backtracking needed!

**Space complexity:**

- CYK table: **O(n²)** space
- Each cell stores set of variables

## Theoretical Importance

### Why CNF Matters in Theory

**1. Canonical Form:**

- Every CFL has a CNF representation
- Provides standard form for comparisons
- Simplifies theoretical proofs

**2. Decidability Results:**

- CNF enables decidable questions
- Membership problem solvable in polynomial time
- Emptiness problem becomes easier

**3. Language Properties:**

- CNF reveals structural properties
- Binary tree structure
- Bounded derivation length

**4. Equivalence Proofs:**

- Easier to prove grammar equivalence
- Standard form reduces cases to consider

### Limitations

**What CNF cannot do:**

✗ **Include ε easily** - must use special handling ✗ **Preserve original structure** - introduces many new variables ✗ **Maintain readability** - often much larger than original ✗ **Show semantic structure** - optimized for parsing, not understanding

**When not to use CNF:**

- Human-readable grammars
- Grammar-based code generation
- When structure matters (programming language syntax)
- Small grammars where conversion overhead isn't worth it

## Greibach Normal Form (Brief Overview)

### Alternative Normal Form

**Greibach Normal Form (GNF):** Every production has form:

- **A → aα** where a is a terminal and α is a string of variables

**Comparison with CNF:**

|Aspect|CNF|GNF|
|---|---|---|
|Form|A → BC or A → a|A → aα|
|Terminals|At end of derivation|At start of every step|
|Variables|0, 1, or 2 per body|Any number|
|Parse trees|Binary|Variable branching|
|Main use|CYK algorithm|PDA construction|

**GNF Properties:**

- Every derivation step generates a terminal
- String of length n requires exactly n derivation steps
- Useful for constructing PDAs from grammars
- More complex conversion process than CNF

### Normal Form Conversion Summary

|Step|Input|Process|Output|Language Change|
|---|---|---|---|---|
|1|Original CFG|Eliminate useless symbols|Cleaner grammar|None (L preserved)|
|2|Clean grammar|Eliminate ε-productions|No ε-productions|L → L - {ε}|

### Production Form Evolution

```
Original:
  S → ASB | ε | b | A
  A → aA | B | ε
  
After useless elimination:
  S → ASB | ε | b | A
  A → aA | B | ε
  
After ε-elimination:
  S → ASB | SB | AS | S | b | A
  A → aA | a | B
  
After unit elimination:
  S → ASB | SB | AS | b | aA | a
  A → aA | a
  
After terminal arrangement:
  S → ASB | SB | AS | b | UₐA | a
  A → UₐA | a
  Uₐ → a
  
After body breaking (CNF):
  S → AD₁ | SB | AS | b | UₐA | a
  A → UₐA | a
  D₁ → SB
  Uₐ → a
```
